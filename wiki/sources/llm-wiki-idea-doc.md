# LLM 위키 아이디어 문서

## 한줄 요약

> LLM이 점진적으로 영속적인 위키를 구축·유지관리하여, RAG의 "축적 없음" 문제를 해결하는 개인 지식 관리 패턴을 제안한다.

## 핵심 내용

### 원문 파트 — 패턴 설명

기존 RAG(NotebookLM, ChatGPT 파일 업로드 등)는 매 질문마다 원시 문서에서 관련 청크를 검색하여 답변을 생성한다. 지식이 축적되지 않고 매번 처음부터 재발견해야 한다는 구조적 한계가 있다.

LLM 위키 패턴은 이와 다르다:
- 새 소스가 들어오면 LLM이 읽고, 핵심을 추출한 뒤 기존 위키에 **통합**한다
- 엔티티 페이지를 업데이트하고, 모순을 기록하고, 종합 분석을 강화한다
- 지식은 한 번 컴파일되고 **최신 상태로 유지**된다

3레이어 아키텍처:
1. **원시 소스(Raw)** — 불변. LLM은 읽기만 함
2. **위키(Wiki)** — LLM이 소유. 생성·수정·상호참조 유지
3. **스키마(Schema)** — CLAUDE.md 등. 구조·규칙·워크플로우 정의

3대 오퍼레이션:
1. **수집(Ingest)** — 소스 읽기 → 요약 → 위키 전반 업데이트
2. **질의(Query)** — 인덱스 탐색 → 관련 페이지 읽기 → 종합 답변
3. **점검(Lint)** — 모순, 고아 페이지, 빈약한 페이지 등 건강 상태 점검

적용 분야: 개인 학습, 리서치, 독서, 비즈니스/팀 위키, 경쟁 분석 등.

정신적 원류는 바네바 부시의 [[memex]](1945) — 문서 간 연상적 경로를 가진 개인 지식 저장소.

### 역자 주석 파트 — 실전 가이드 (10개 항목)

1. **Obsidian ↔ LLM 연동**: 볼트 루트를 에이전트 작업 디렉토리로 설정. `.obsidian/` 보호 필수. Linter·Templater 플러그인 추천
2. **스키마 필수 항목**: 디렉토리 구조, 페이지 템플릿, 네이밍 규칙, 수집 워크플로우, 금지 사항
3. **[[mcp]] 서버**: 위키 검색을 MCP로 노출하면 LLM이 네이티브 도구로 직접 검색. 50페이지 이상이면 검토
4. **한국어 운영**: 파일명 영어 kebab-case, 제목 한국어, 태그 한영 병기. 형태소 분석 없으면 검색에 한계
5. **소스 수집 팁**: YouTube(yt-dlp), PDF(marker), 트위터(Thread Reader), 팟캐스트(Whisper)
6. **비용 감각**: 소스 1건 수집 = 수만~십수만 토큰. Max Plan이면 큰 문제 없음
7. **git 전략**: 수집 단위 커밋, 실험적 작업은 브랜치, `.obsidian/workspace.json` gitignore
8. **RAG 병행**: 위키 = 1차 지식 레이어, RAG = 원문 검증 2차 레이어. 대체가 아닌 보완 관계
9. **세션 간 컨텍스트**: 위키가 해결하는 진짜 문제. 세션 시작 시 index.md + log.md 읽기로 자동화
10. **점검 프롬프트**: 7개 항목 체크리스트 예시 제공. 스키마에 넣어두면 "lint 돌려줘"로 호출 가능

## 주요 인사이트

- **위키는 복리로 축적되는 산출물**이라는 프레이밍이 핵심. 소스를 추가할수록, 질문을 할수록 위키가 풍부해진다.
- 사람은 기록 관리(상호참조, 요약 최신화, 일관성 유지) 부담 때문에 위키를 포기하지만, LLM은 이 유지관리를 거의 0 비용으로 수행한다.
- "좋은 답변은 위키에 다시 저장할 수 있다" — 탐색 활동 자체도 지식 축적의 원천이 된다.
- RAG와의 관계를 대체가 아닌 **보완**으로 보는 역자 주석의 관점이 실용적이다.

## 인용할 만한 구절

> "Obsidian은 IDE이고, LLM은 프로그래머이고, 위키는 코드베이스입니다."

> "사람은 유지관리 부담이 가치보다 빠르게 증가하기 때문에 위키를 포기합니다. LLM은 지루해하지 않고, 상호 참조 업데이트를 잊지 않으며, 한 번에 15개 파일을 수정할 수 있습니다."

> "위키가 '컴파일된 지식'이라면, RAG는 '원문 검증 도구'인 셈입니다." — 역자 주석
