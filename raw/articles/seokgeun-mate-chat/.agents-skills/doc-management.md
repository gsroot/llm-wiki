# 문서 관리

Mate Chat 프로젝트의 문서 작성 및 코드-문서 동기화를 위한 통합 스킬.

---

## 관련 시스템

**Slash Command:** 없음 (자동 트리거)
- "문서 작성", "문서 업데이트" 키워드 감지 시 자동 트리거

**Sub-Agent:** 없음 (독립적 운영)
- 이 스킬은 3단계 문서 작성 워크플로우를 독립적으로 실행

**Agent Skill:** `doc-management` (이 스킬 - 자동 트리거)
- 트리거 1: 사용자가 새 문서 작성 요청 시
- 트리거 2: API 변경, 스키마 변경, 기능 추가 등 코드 변경 감지 시
- 역할: 구조화된 문서 작성 워크플로우 제공 및 코드-문서 동기화

**관련 스킬:**
- `api-consistency`: API 변경 시 문서 업데이트 필요
- `migration-safety`: 데이터베이스 변경 시 문서 업데이트 필요
- `feature-workflow`: 새 기능 추가 시 문서 업데이트 포함

---

## 사용 시나리오

이 스킬은 두 가지 주요 시나리오에서 사용됩니다:

### 시나리오 1: 새 문서 작성
사용자가 새로운 문서를 작성하려고 할 때 (설계 문서, 기술 사양, 제안서, PRD, RFC 등)

### 시나리오 2: 코드 변경 후 문서 동기화
코드 변경 후 관련 문서를 업데이트해야 할 때 (API 추가, 스키마 변경, 기능 수정 등)

---

## 시나리오 1: 새 문서 작성 워크플로우

사용자가 "문서 작성", "설계 문서", "기술 사양" 등을 언급하면 이 워크플로우를 제안합니다.

### 워크플로우 제안

사용자에게 구조화된 3단계 문서 작성 워크플로우를 제안:

1. **Context Gathering (컨텍스트 수집)** - 필요한 모든 정보 수집
2. **Refinement & Structure (정제 및 구조화)** - 섹션별로 문서 작성 및 개선
3. **Reader Testing (독자 테스트)** - 문서가 실제로 작동하는지 테스트

사용자가 워크플로우를 수락하면 진행, 거절하면 자유형식으로 작업합니다.

### Stage 1: Context Gathering

**목표**: 문서 작성에 필요한 모든 컨텍스트 수집

#### 초기 질문

다음 질문으로 시작:
1. 어떤 유형의 문서인가요? (기술 사양, 설계 문서, 제안서 등)
2. 주요 독자는 누구인가요?
3. 이 문서를 읽은 후 어떤 영향을 기대하나요?
4. 따라야 할 템플릿이나 형식이 있나요?
5. 알아야 할 제약사항이나 컨텍스트가 있나요?

#### 정보 수집

사용자에게 모든 관련 컨텍스트를 자유롭게 제공하도록 요청:
- 프로젝트/문제 배경
- 관련 팀 논의나 문서
- 대안 솔루션을 사용하지 않는 이유
- 조직 컨텍스트
- 타임라인 제약사항
- 기술 아키텍처나 의존성
- 이해관계자 우려사항

#### 명확화 질문

초기 정보 수집 후 5-10개의 명확화 질문 생성:
- 컨텍스트에서 불분명한 부분
- 누락된 중요 정보
- 에지 케이스나 트레이드오프

사용자가 충분한 컨텍스트를 제공할 때까지 반복합니다.

### Stage 2: Refinement & Structure

**목표**: 섹션별로 문서를 작성하고 반복적으로 개선

#### 문서 구조 결정

문서 유형에 따라 3-5개 섹션 제안:
- 설계 문서: 개요, 배경, 제안, 대안, 영향 분석
- API 문서: 개요, 엔드포인트, 요청/응답, 에러 처리
- 데이터베이스 스키마: 테이블 목적, 컬럼 설명, 관계, 인덱스

#### 각 섹션마다 반복

**1. 명확화 질문** - 섹션에 포함할 내용에 대한 5-10개 질문

**2. 브레인스토밍** - 섹션 복잡도에 따라 5-20개 항목 제안

**3. 큐레이션** - 사용자가 유지/제거/결합할 항목 선택

**4. 갭 체크** - 누락된 중요 사항 확인

**5. 초안 작성** - 선택된 내용으로 섹션 작성

**6. 반복 개선** - 사용자 피드백에 따라 수술적 편집 (전체 재작성 X)

#### 품질 확인

3회 연속 큰 변경 없이 반복되면 축약 가능한 내용 확인.

모든 섹션 완료 후 전체 문서 검토:
- 섹션 간 흐름 및 일관성
- 중복 또는 모순
- 일반적인 문구 제거
- 모든 문장이 가치를 전달하는지

### Stage 3: Reader Testing

**목표**: 문서가 실제로 독자에게 작동하는지 검증

#### 독자 질문 예측

독자가 이 문서를 발견할 때 물어볼 5-10개 질문 생성.

#### 테스트 수행

가능하면 sub-agent를 사용하여 자동 테스트, 불가능하면 사용자가 수동 테스트.

각 질문에 대해:
- 답변이 정확한지
- 모호하거나 불분명한 부분이 없는지
- 문서가 가정하는 사전 지식이 무엇인지

#### 추가 체크

- 모호하거나 불분명한 내용
- 가정된 사전 지식
- 내부 모순이나 불일치

#### 반복

문제가 발견되면 Stage 2로 돌아가 해당 섹션 수정.

Reader Testing을 통과하면 문서 완료.

## 시나리오 2: 코드 변경 후 문서 동기화

API 변경, 스키마 변경, 기능 추가 등 코드 변경 감지 시 자동 트리거.

### 1단계: 변경사항 식별

변경된 파일 경로를 분석하여 업데이트할 문서 식별.

**파일 경로 기반 감지**:
```
app/api/v1/endpoints/*.py → docs/14-api-design.md, AGENTS.md
app/models/*.py → docs/13-database-schema.md
app/services/auth*.py → docs/15-auth-system.md
app/websocket/*.py → docs/16-realtime-websocket.md
pyproject.toml → AGENTS.md (기술 스택)
```

**코드 패턴 기반 감지**:
- `@router.*` 데코레이터 → API 엔드포인트 개수 업데이트
- `class.*Base` → 데이터베이스 테이블 개수 업데이트
- `def test_` → 테스트 개수 업데이트

상세한 매핑은 [doc-mapping.md](./references/doc-mapping.md)를 참조하세요.

### 2단계: 문서 업데이트 리마인더

사용자에게 업데이트가 필요한 문서와 섹션을 명확히 알립니다:

```
코드 변경 감지: app/api/v1/endpoints/chats.py

업데이트 필요:
1. docs/14-api-design.md - Chat 엔드포인트 섹션
2. AGENTS.md - 엔드포인트 개수 (83 → 84)
3. AGENTS.md - 최종 업데이트 날짜

문서 업데이트를 도와드릴까요?
```

### 3단계: 문서 업데이트 지원

사용자가 승인하면:
1. [update-guide.md](./references/update-guide.md)의 템플릿 사용
2. 관련 섹션 업데이트
3. 메타데이터 갱신 (개수, 완료율, 날짜)
4. Stage 1 워크플로우 적용 (필요시)
