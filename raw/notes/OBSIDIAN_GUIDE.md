# Obsidian 사용 가이드

부제: Notion 사용자도 바로 적응하는 실무형 입문서

## 1. Obsidian을 한 줄로 설명하면

Obsidian은 **내 컴퓨터 폴더(vault)에 Markdown 노트를 저장하고, 노트끼리 링크해서 지식을 연결하는 도구**다.

핵심은 딱 네 가지다.

- 파일이 내 손에 남는다.
- 노트가 링크로 연결된다.
- 속성(Properties)으로 구조를 붙일 수 있다.
- 검색으로 다시 꺼내 쓰기 쉽다.

그래서 Obsidian은 단순한 메모앱이라기보다 **개인 또는 팀의 지식 운영 시스템**에 가깝다.

## 2. 언제 쓰면 좋고, 언제 덜 맞는가

## 잘 맞는 경우

- 문서가 앱 안에 갇히지 않고 파일로 남았으면 좋겠다.
- 노트 사이 관계를 링크로 엮고 싶다.
- 오프라인에서도 작업하고 싶다.
- 내 방식대로 폴더, 속성, 템플릿을 설계하고 싶다.
- 회의 기록, 프로젝트 정리, 학습 노트, 아이디어 축적을 한곳에 모으고 싶다.

## 덜 맞는 경우

- 여러 사람이 동시에 한 문서를 실시간 공동편집해야 한다.
- 처음부터 완성된 협업 데이터베이스 UI가 필요하다.
- 구조를 스스로 설계하는 것보다 정해진 틀을 선호한다.

내 의견은 명확하다.
**개인 지식관리나 실무 기록 축적에는 Obsidian이 꽤 강력하다.**
반면, **동시 협업 문서 편집 자체가 핵심이면 Notion/Google Docs가 더 편한 경우가 많다.**

## 3. 설치 후 가장 먼저 할 일

설치가 끝났으면 바로 기능 탐험부터 하지 말고 아래 순서로 시작하면 된다.

### 1단계. 첫 Vault 만들기

Vault는 Obsidian이 노트를 저장하는 폴더다.
처음에는 하나만 만들면 된다.

권장 예시:

- `Work Vault`
- `Personal Vault`
- 또는 그냥 `Main Vault`

초반에는 vault를 여러 개 만드는 것보다 **하나로 시작해서 운영 감각을 잡는 게 낫다.**

### 2단계. 폴더는 최소한으로 만든다

초기 권장 폴더 구조:

- `00 Inbox`
- `10 Projects`
- `20 Knowledge`
- `30 Meetings`
- `40 Daily`
- `Templates`
- `Assets`
- `90 Archive`

이 정도면 충분하다.
처음부터 폴더를 20개 이상 만들면 거의 확실하게 지친다.

### 3단계. 코어 플러그인만 먼저 쓴다

처음에는 기본 기능 위주로 쓰는 걸 추천한다.
먼저 익히면 좋은 것:

- Backlinks
- Search
- Quick switcher
- Page preview
- Templates
- Daily notes
- Outline
- Tags view
- Properties view

커뮤니티 플러그인은 나중에 넣어도 늦지 않다.

## 4. 첫 5분 안에 해야 하는 것

### 새 노트 3개를 만든다

예시:

- `프로젝트 허브`
- `2026-04-07 회의`
- `오늘 메모`

### 노트끼리 링크한다

`프로젝트 허브`에 아래처럼 적어본다.

```md
# 프로젝트 허브

## 관련 문서
- [[2026-04-07 회의]]
- [[오늘 메모]]
```

이 순간부터 Obsidian은 그냥 파일 저장소가 아니라 **연결된 문서 공간**이 된다.

### 회의 노트 안에서 다시 허브로 연결한다

```md
# 2026-04-07 회의

관련 프로젝트: [[프로젝트 허브]]
```

이렇게 양방향 흐름을 만들면, 나중에 백링크에서 문맥이 살아난다.

## 5. 기본 작성법

Obsidian은 Markdown을 사용한다.
처음에는 아래 정도만 알면 충분하다.

```md
# 제목 1
## 제목 2
### 제목 3

- 글머리표
- 글머리표

1. 번호 목록
2. 번호 목록

- [ ] 할 일
- [x] 완료한 일

**굵게**
*기울임*

`인라인 코드`
```

코드 블록 예시:

```sql
SELECT *
FROM sales
WHERE dt >= '2026-04-01';
```

팁 하나만 말하면,
**문법을 외우려고 하지 말고 템플릿으로 반복하면서 몸에 붙이는 게 훨씬 빠르다.**

## 6. 링크 중심으로 정리하는 법

Obsidian의 핵심은 링크다.
폴더는 저장 위치고, 진짜 구조는 링크에서 나온다.

## 링크 기본

```md
[[프로젝트 허브]]
```

표시 이름을 바꾸고 싶으면:

```md
[[프로젝트 허브|결제 개선 프로젝트]]
```

헤더로 바로 링크할 수도 있다.

```md
[[프로젝트 허브#다음 액션]]
```

## 링크를 왜 쓰는가

예를 들어 회의 노트 100개가 생겨도,
각 회의 노트가 프로젝트 허브를 가리키고 있으면 프로젝트 허브 하나에서 맥락을 다시 모을 수 있다.

추천 패턴:

- 회의 노트 → 프로젝트 허브 링크
- 이슈 노트 → 서비스/기능 노트 링크
- 의사결정 노트 → 관련 회의 / 요구사항 / 구현 노트 링크
- 데일리 노트 → 그날 작업한 프로젝트 노트 링크

## 백링크를 같이 보라

Obsidian은 현재 노트를 가리키는 다른 노트들을 백링크로 보여준다.
이 기능 덕분에 “내가 이 주제를 어디서 또 언급했지?”를 금방 찾을 수 있다.

실무에서는 이게 엄청 크다.
문서를 직접 찾아 들어가지 않아도, 관련 문맥이 반대로 모여든다.

## 7. 파일명 규칙은 초반에 잡아라

권장 규칙:

- 회의 노트: `2026-04-07 결제 개선 회의`
- 일일 노트: `2026-04-07`
- 프로젝트 노트: `프로젝트 - 결제 개선`
- 의사결정 노트: `Decision - 결제 실패 재시도 정책`
- 개념 노트: `개념 - 리텐션 정의`

규칙을 잡는 이유는 단순하다.

- Quick switcher에서 잘 찾힌다.
- 검색 품질이 좋아진다.
- 폴더보다 파일명만으로도 정렬과 구분이 된다.

주의할 점:

- 파일명은 너무 길지 않게
- 특수문자 남발하지 않기
- 날짜는 가능하면 `YYYY-MM-DD` 통일

## 8. Properties와 Tags를 어떻게 나눠 쓸까

이 둘을 섞어 쓰기 시작하면 구조가 훨씬 좋아진다.
다만 역할을 구분해야 한다.

## Properties는 “구조화된 정보”

추천 최소 속성:

- `type`
- `status`
- `date`
- `owner`
- `tags`

예시:

```md
---
type: meeting
status: done
date: 2026-04-07
owner: 석근
tags:
  - meeting
  - billing
---
```

언제 Properties를 쓰나?

- 정해진 키로 반복 입력할 때
- 나중에 정렬/필터링/조회하고 싶을 때
- Bases 같은 구조화 뷰로 확장할 가능성이 있을 때

## Tags는 “짧고 반복적인 분류”

예시:

- `#meeting`
- `#project`
- `#billing`
- `#inbox/to-read`

언제 Tags를 쓰나?

- 같은 성격의 노트를 빠르게 묶고 싶을 때
- 검색과 태그 뷰에서 재탐색하고 싶을 때
- 엄격한 스키마까지는 필요 없을 때

## Links는 “맥락 연결”

예시:

- `[[프로젝트 - 결제 개선]]`
- `[[개념 - 리텐션 정의]]`
- `[[2026-04-07 결제 개선 회의]]`

언제 Links를 쓰나?

- 문서와 문서의 관계를 남기고 싶을 때
- 주제 간 연결을 만들고 싶을 때
- 나중에 백링크로 문맥을 보고 싶을 때

## 내가 추천하는 기준

- **반복 분류**는 태그
- **고정된 필드**는 속성
- **문맥 연결**은 링크

이 기준만 지켜도 구조가 거의 안 망가진다.

## 9. 검색을 잘 쓰면 정리가 쉬워진다

정리를 열심히 하는 사람보다,
**검색을 잘 설계하는 사람이 나중에 더 편하다.**

기본 예시:

```text
"결제 오류"
meeting OR 회의
회의 -개인
```

자주 쓸 만한 예시:

```text
tag:#meeting
file:2026-04
path:"30 Meetings"
```

의미:

- `"결제 오류"` : 정확히 그 문구가 들어간 노트 검색
- `meeting OR 회의` : 둘 중 하나라도 포함한 노트 검색
- `회의 -개인` : 회의는 포함하지만 개인은 제외
- `tag:#meeting` : meeting 태그가 있는 노트 검색
- `file:2026-04` : 파일명에 2026-04가 들어간 파일 검색
- `path:"30 Meetings"` : 해당 폴더 경로 아래 파일 검색

검색은 결국 나중에 다시 찾는 기술이다.
그래서 제목, 태그, 속성, 링크를 일정하게 쓰는 습관이 중요하다.

## 10. Quick switcher와 Command palette는 빨리 익혀라

## Quick switcher

노트를 빨리 열고, 없으면 바로 새 노트를 만들 수 있다.

추천 사용법:

- 프로젝트명 일부 입력해서 바로 이동
- 날짜 입력해서 데일리/회의 노트 열기
- 비슷한 이름의 문서를 빠르게 비교

## Command palette

명령을 이름으로 바로 실행한다.
예를 들면 이런 작업들이 빨라진다.

- 템플릿 삽입
- 오늘 데일리 노트 열기
- 백링크 패널 열기
- 특정 기능 켜기/끄기

초반에 마우스로만 쓰면 Obsidian의 장점 절반을 버리는 셈이다.
키보드 흐름을 빨리 익히는 쪽이 훨씬 낫다.

## 11. Templates로 반복 작업을 없애라

Obsidian이 편해지는 순간은 빈 문서에서 시작하지 않게 되는 순간이다.
그 역할을 템플릿이 한다.

## 추천 템플릿 1. 회의 노트

```md
---
type: meeting
status: done
date: {{date:YYYY-MM-DD}}
tags:
  - meeting
---

# {{title}}

## 목적

## 논의 내용

## 결정 사항

## 액션 아이템
- [ ]

## 관련 문서
- [[프로젝트 - ]]
```

## 추천 템플릿 2. 프로젝트 허브

```md
---
type: project
status: active
owner:
tags:
  - project
---

# {{title}}

## 목표

## 현재 상태

## 핵심 링크
- [[회의]]
- [[요구사항]]
- [[결정사항]]

## 다음 액션
- [ ]
```

## 추천 템플릿 3. 참고/학습 노트

```md
---
type: reference
status: active
tags:
  - reference
---

# {{title}}

## 한 줄 요약

## 핵심 내용

## 실무 적용 포인트

## 관련 노트
- [[ ]]
```

## 12. Daily notes는 기록 습관을 만든다

데일리 노트는 “오늘 뭘 했는지”, “무엇을 배웠는지”, “어디서 막혔는지”를 한 장으로 남기는 데 좋다.

추천 데일리 노트 템플릿:

```md
---
type: daily
date: {{date:YYYY-MM-DD}}
tags:
  - daily
---

# {{date:YYYY-MM-DD}}

## 오늘 할 일
- [ ]

## 작업 로그
- 

## 배운 점
- 

## 연결할 노트
- [[프로젝트 - ]]
- [[회의 - ]]
```

데일리 노트의 좋은 점은 단순하다.

- 매일 무엇을 했는지 남는다.
- 프로젝트와 회의 노트에 자연스럽게 링크가 걸린다.
- 나중에 “언제 이 이슈를 봤지?”를 복기하기 쉬워진다.

## 13. Canvas는 시각 정리용, Bases는 구조화용

둘 다 좋다. 그런데 역할이 다르다.

## Canvas

이럴 때 좋다.

- 아이디어를 마인드맵처럼 펼치고 싶을 때
- 프로젝트 관계도를 그리고 싶을 때
- 문서/이미지/웹페이지를 한 화면에 묶고 싶을 때

한마디로,
**생각을 펼칠 때** 좋다.

## Bases

이럴 때 좋다.

- 회의 노트를 목록/카드 형태로 보고 싶을 때
- 프로젝트 문서를 상태별로 모아보고 싶을 때
- properties를 기준으로 정렬/필터링하고 싶을 때

한마디로,
**정리된 데이터를 조회할 때** 좋다.

## 내 추천 순서

1. 먼저 노트 + 링크 + 속성부터 익힌다.
2. 그다음 필요할 때 Bases를 붙인다.
3. 시각적으로 정리하고 싶을 때 Canvas를 쓴다.

초반부터 Bases에 꽂히면 구조를 만들다 끝나고,
Canvas에 꽂히면 예쁘게 그리다 끝난다.
먼저 노트가 쌓여야 둘 다 힘을 발휘한다.

## 14. Sync와 백업은 가볍게 보면 안 된다

Obsidian은 파일 기반이라서 백업과 동기화 전략이 중요하다.

## 원칙 1. 동기화 전략은 하나만 메인으로 잡아라

예를 들면:

- Obsidian Sync
- iCloud
- Dropbox
- OneDrive
- Git

이 중 무엇을 쓸지는 상황마다 다르지만,
**여러 전략을 섞다가 충돌나는 경우가 제일 피곤하다.**

## 원칙 2. `.obsidian` 폴더도 같이 관리할지 결정하라

이 폴더에는 설정, 테마, 단축키, 플러그인 정보가 들어간다.

- 여러 기기에서 같은 환경을 쓰고 싶다면 같이 동기화/백업
- 기기마다 설정을 다르게 쓰고 싶다면 분리 고려

## 원칙 3. 중요한 vault는 정기 백업하라

특히 아래 중 하나면 백업은 필수다.

- 회사 업무 문서
- 장기 연구/학습 노트
- 여러 기기에서 동시에 쓰는 vault
- 플러그인을 많이 쓰는 vault

## 15. 커뮤니티 플러그인은 늦게 넣는 게 맞다

커뮤니티 플러그인은 강력하다.
하지만 강력한 만큼 주의도 필요하다.

내 추천은 이렇다.

### 처음 1주일
- 커뮤니티 플러그인 없이 사용
- 기본 기능 흐름 익히기

### 그다음
- 불편이 명확할 때만 플러그인 추가
- 한 번에 여러 개 말고 하나씩
- 문제가 생기면 원인 추적 가능하게 유지

### 팀/업무 환경
- 검증되지 않은 플러그인 무분별 도입 금지
- 데이터 접근 권한과 유지보수 가능성 확인
- 누가 왜 쓰는지 기록 남기기

플러그인이 많아질수록 생산성이 늘 수도 있지만,
관리 비용도 같이 늘어난다.
초보자일수록 이 함정을 과소평가한다.

## 16. 실무 추천 운영 방식

여기부터가 진짜 중요하다.
문서를 예쁘게 만드는 것보다 **운영 규칙**이 중요하다.

## 추천 규칙 1. 프로젝트마다 허브 노트 하나

예시:

- `프로젝트 - 결제 개선`
- `프로젝트 - BI 지표 표준화`

허브 노트 안에 모은다.

- 목표
- 관련 회의
- 요구사항
- 결정사항
- 할 일
- 참고 링크

그러면 프로젝트 관련 문서가 흩어져도 중심점이 생긴다.

## 추천 규칙 2. 회의는 무조건 프로젝트에 연결

회의 노트는 단독으로 두지 말고,
최소한 아래 둘 중 하나에는 연결한다.

- 프로젝트 노트
- 이슈/기능 노트

그래야 나중에 회의록이 고아 문서가 안 된다.

## 추천 규칙 3. 의사결정은 따로 남겨라

회의 내용 전체보다 중요한 건 결정이다.
그래서 아래처럼 분리해도 좋다.

- `Decision - 결제 실패 시 재시도 3회`
- `Decision - 리텐션 기준 D1/D7/D30 확정`

이런 노트는 나중에 정책 변경 이력이나 근거 찾기에 아주 유용하다.

## 추천 규칙 4. 태그는 적게, 속성은 얇게

초기 권장:

- 태그 5~10개 안에서 시작
- 속성도 4~5개 정도만 유지

처음부터 태그 50개 만들고 속성 15개 만들면 유지가 안 된다.
작게 시작해서 필요한 것만 늘리는 편이 맞다.

## 17. Notion 사용자라면 이렇게 이해하면 쉽다

아주 거칠게 말하면:

- Notion은 블록 중심으로 문서를 조립하는 느낌
- Obsidian은 파일/노트 중심으로 지식을 연결하는 느낌

따라서 적응 포인트는 이렇다.

### Notion식 사고
- 페이지를 만든다
- 데이터베이스를 만든다
- 보기 좋게 배치한다

### Obsidian식 사고
- 노트를 만든다
- 링크로 연결한다
- 속성으로 얇게 구조를 준다
- 검색과 백링크로 다시 꺼낸다

즉,
**처음부터 완성형 데이터베이스를 만들려고 하지 말고, 먼저 노트와 링크를 쌓아라.**
나중에 Bases가 그 위에 얹히면 된다.

## 18. 흔한 실수

## 실수 1. 폴더를 너무 많이 만든다

처음엔 다 정교해 보인다.
한 달 지나면 대부분 귀찮아진다.

## 실수 2. 태그를 너무 많이 만든다

태그 체계가 복잡해질수록 입력이 귀찮아지고 일관성이 깨진다.

## 실수 3. 플러그인을 너무 빨리 깐다

새로워 보여서 이것저것 깔기 시작하면,
어느 순간 기본 사용법보다 설정창을 더 오래 보게 된다.

## 실수 4. 링크를 안 건다

그럼 Obsidian이 아니라 그냥 Markdown 폴더가 된다.

## 실수 5. 템플릿 없이 매번 빈 페이지에서 시작한다

이건 장기적으로 거의 못 버틴다.
반복 문서는 템플릿으로 줄이는 게 맞다.

## 실수 6. 파일명 규칙이 없다

나중에 검색이 지저분해지고, 회고할 때 고생한다.

## 19. 1주차 실행 체크리스트

### Day 1
- Vault 1개 만들기
- 기본 폴더 구조 만들기
- 프로젝트 허브 노트 1개 만들기

### Day 2
- 회의 노트 템플릿 만들기
- 실제 회의 하나를 Obsidian에 기록하기

### Day 3
- 데일리 노트 템플릿 만들기
- 하루 작업 로그 남기기

### Day 4
- 링크 10개 이상 걸어보기
- 백링크 패널 확인해보기

### Day 5
- `type`, `status`, `date`, `tags` 속성 넣어보기
- 태그 3~5개만 정해서 사용하기

### Day 6
- 검색 예시 5개 직접 써보기
- Quick switcher만으로 노트 열어보기

### Day 7
- 안 쓰는 구조 삭제하기
- 꼭 필요한 것만 남기기
- 커뮤니티 플러그인 도입이 정말 필요한지 판단하기

## 20. 최소 운영안 요약

정말 바쁘면 이것만 기억하면 된다.

1. Vault 하나로 시작한다.
2. 폴더는 적게 만든다.
3. 노트는 링크로 연결한다.
4. 속성은 적게, 태그는 더 적게 쓴다.
5. 템플릿과 데일리 노트를 먼저 만든다.
6. 검색과 Quick switcher를 적극적으로 쓴다.
7. 플러그인은 늦게 넣는다.
8. 백업은 꼭 한다.

## 21. 마지막 한마디

Obsidian을 잘 쓰는 사람은 복잡한 시스템을 만든 사람이 아니다.
**작고 일관된 규칙을 오래 지킨 사람**이다.

처음부터 완벽한 지식관리 시스템을 만들 생각은 버리는 게 좋다.
대신 아래만 하자.

- 오늘 노트를 남긴다.
- 관련 노트를 링크한다.
- 내일 다시 찾을 수 있게 제목과 태그를 정리한다.

그걸 반복하면, 어느 순간 vault가 그냥 메모 폴더가 아니라 **내가 일하고 생각한 흔적이 쌓인 운영 체계**가 된다.
